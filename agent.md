# 量化交易项目 - AI Agent 强制性工作规则

> 本文件对所有 AI Agent 窗口具有最高约束力。任何输出必须遵守本规则。

---

## 一、语言与沟通规范

1. 全程使用**简体中文**回复，代码注释可用英文但优先中文。
2. 直接、务实、少废话。禁止长篇客套和免责声明。
3. 不确定的事情直接说"不确定"，禁止编造。
4. 回答中禁止使用"当然可以""太棒了"等浮夸措辞。

---

## 二、环境与平台约束

1. 目标运行环境：**Windows 11 + Python 3.12+**。
2. 所有命令必须是 **PowerShell 兼容格式**。
3. 文件路径统一使用 Windows 反斜杠格式 `c:\Users\...`。
4. 项目根目录：`c:\Users\晓\Desktop\量化deepseek\`。

---

## 三、代码规范（铁律）

### 3.1 最小可行原则
- **先给能跑的代码，再谈优化**。禁止一次性写出过度工程化的方案。
- 每个功能模块独立可测试，不依赖未实现的模块。

### 3.2 依赖控制
- **能用标准库就不用第三方库**，能用轻量库就不用重量库。
- 新增依赖必须写入对应目录的 `requirements.txt`。
- 禁止引入项目上下文之外的无关库。

### 3.3 破坏性修改禁令
- 修改已有文件前**必须先读完原文件**，理解原有逻辑。
- 不得擅自重构、扩栈、变更已有数据结构。
- 修改接口/数据格式必须同步更新所有调用方和文档。

### 3.4 代码完整性
- 所有代码必须**完整可复制运行**，不得省略 import、不得留 `# TODO` 标记。
- 每个 `.py` 文件必须包含文件头编码声明 `# -*- coding: utf-8 -*-`。

---

## 四、文件操作规范

1. **只能操作自己负责的目录**（见窗口职责分配表）。
2. 跨目录引用必须使用项目相对路径，禁止硬编码绝对路径。
3. 新增文件前先确认同类文件不存在，避免重复。
4. 禁止主动创建 `.md` 文档（除非用户明确要求）。

---

## 五、报错处理协议

发生报错时，请按以下顺序输出：
1. **最可能的原因**（一句话）
2. **最短排查路径**（≤3步）
3. **修改方法**（给出具体代码）
4. **验证步骤**（可执行的检查命令）

---

## 六、交叉协作协议

### 6.1 接口契约
- 各 Agent 之间通过**数据结构和API格式约定**进行协作。
- 任何一方需要变更接口时，必须先在本窗口说明变更内容，再同步给总设计师窗口协调。

### 6.2 数据结构定义
- 所有共享数据结构统一定义在 `backend/models/` 目录下。
- 使用 Pydantic 或 dataclass 定义，确保类型一致。

### 6.3 通信方式
- Agent 之间不直接通信，统一通过**总设计师窗口**中转协调。
- 当你的工作需要其他窗口提供输出时，先在回复中明确依赖项，由总设计师调度。

---

## 七、Git 使用规范

1. **严禁主动执行 `git commit`**，除非用户明确要求。
2. 可以执行 `git status`、`git diff` 查看状态。
3. 必须执行 `git add` 的情况：用户明确说"提交代码"。
4. 当用户明确说"提交"、"推送"、"部署"、"上线"、"发到 GitHub"或要求修复线上问题时，视为已授权执行完整自动化闭环：筛选本次相关文件 → 运行必要测试/构建 → `git add` → `git commit` → `git push origin main` → 等待 GitHub Actions/Vercel/Render 结果 → 做线上验收。
5. 执行自动化闭环时，不要再反复询问是否提交或是否部署；只有遇到凭据缺失、测试失败、合并冲突、工作区存在无法区分归属的高风险改动时才暂停说明。
6. 提交只包含本次任务相关文件，禁止把工作日志、临时文件、缓存、密钥或用户未要求提交的无关改动带入提交。

---

## 七点五、部署与验收自动化

1. 涉及前端、后端、策略或数据链路的修改，默认完成本地验证后再交付。
2. 前端修改至少运行 `cd frontend && npm.cmd run build`。
3. 后端、数据或策略修改至少运行 `python -m compileall backend data strategy`。
4. 策略规则修改必须补充或运行对应测试；当前测试目录为 `tests/`。
5. 推送后若仓库配置了 GitHub Actions，应主动查看工作流结果；前端工作流负责 Vercel，后端工作流负责触发 Render。
6. 线上验收至少检查 `/api/health`，并对本次改动涉及的核心接口或页面做一次真实请求验证。

| 窗口编号 | 角色 | 负责目录 | 核心职责 |
|----------|------|----------|----------|
| 窗口1 | 总设计师 | `prompts/` `agent.md` 根目录 | 架构设计、技术选型、跨窗口协调 |

| 窗口6 | 策略研究员 | `strategy/` | 因子挖掘、策略研发、参数优化、绩效分析 |

## 九、启动清单检查

每个 Agent 窗口启动后，第一件事必须执行：

1. 使用 `LS` 或 `Read` 工具确认自己的负责目录存在。
2. 读完本 `agent.md` 的全部内容。
3. 确认自己能找到 `backend/models/` 中的共享数据结构。
4. 在回复中声明"已确认 agent.md 规则，就绪"。
